Scheme Eligibility Advisor + Langfuse Observability

You are working on JanVedha AI, an open-source civic governance platform for Indian municipal bodies. The backend is FastAPI + MongoDB (Beanie ODM). The frontend is Next.js with Tailwind and Recharts. The existing AI features use Google Gemini via LangChain. There is a second repository — RAG-System — which is a fully working hybrid RAG pipeline built with ChromaDB, BM25, RRF fusion, a cross-encoder reranker, and Ollama as the LLM. You will be porting the retrieval architecture from that RAG-System repo into JanVedha and replacing Ollama with Gemini. Everything else in the RAG pipeline — the chunker, embedder (all-MiniLM-L6-v2), vector store, BM25 store, hybrid retriever, and cross-encoder reranker — stays exactly as built.

The Feature: Constituent Rights & Scheme Eligibility Advisor
Ward councillors in India regularly meet constituents who ask which government schemes they qualify for. Currently, councillors either don't know or have to manually look through dozens of scheme documents. The goal is to give them a RAG-powered tool inside the JanVedha dashboard where they type a constituent profile in plain language — something like "BPL family, 3 children, no pucca house, income ₹1.8 lakh per year, SC category, ward 7" — and the system returns a structured eligibility assessment: which schemes this person is eligible for, what documents they need to apply, where to apply, and why the system thinks they qualify, with citations to the actual government scheme documents.
The knowledge base (the document corpus the RAG system retrieves from) must be seeded with real Indian government scheme documents. The priority sources are: PMAY-U 2.0 guidelines (central housing scheme), Ayushman Bharat PM-JAY (health insurance), PM Ujjwala Yojana 2.0 (LPG connections), MGNREGS employment guarantee documents.  a small set of manually written Markdown files with clean, structured eligibility criteria for the top 8-10 schemes will dramatically improve retrieval accuracy compared to relying on dense PDF layouts alone.
The response to the councillor must be structured — not prose. The agent should return which schemes the constituent is eligible for with high/medium/low confidence, which schemes they might be eligible for pending more information (and what information is missing), what documents are needed for each, and the application portal or office. Every eligibility claim must be traceable to a specific retrieved document chunk. The system must refuse to invent eligibility criteria — if the retrieved documents don't contain enough information, it says so.

The Langfuse Integration
Every AI call in this feature — and ideally across the entire JanVedha platform — must be observable through Langfuse. Langfuse must be self-hosted, running as a Docker service alongside JanVedha's existing infrastructure. It cannot be the cloud version because this is a government platform handling civic data. The self-hosted Langfuse instance is free, MIT licensed for core features, and runs in the same Docker network as the backend so traces never leave the internal network.
What Langfuse does for this project across its four capabilities:
Observability — Every scheme eligibility query generates a full trace in Langfuse showing the complete execution tree: how long BM25 retrieval took, how long vector search took, how the reranker changed the chunk ordering, exactly what prompt was sent to Gemini including the constituent profile and retrieved chunks, exactly what Gemini returned, token count, and estimated cost. This trace is linked to the MongoDB audit record for that query via a trace ID stored in both places.
Prompt Management — The eligibility assessment prompt and the system messages are managed in Langfuse's prompt registry rather than hardcoded in YAML files. This means the prompt can be updated, versioned, and tested without a code deployment. A policy officer can review the prompt for correctness. When the team discovers through evaluations that the prompt is underperforming for Tamil-language profiles, a new version is deployed with a single label change in the Langfuse UI. The running application picks up the new prompt without restarting.
Evaluation — Two evaluation signals run automatically. First, an async LLM-as-judge call runs after every response (non-blocking, does not add latency for the user) that scores the response for faithfulness — whether every claim is traceable to the retrieved documents — and for accuracy — whether the eligibility verdict is correct given the scheme criteria. Second, the councillor dashboard shows thumbs up / thumbs down buttons below every scheme response, and those scores are written back to Langfuse as human feedback signals. Traces that score poorly on either dimension are surfaced for review.
Analytics — The Commissioner's dashboard gets a new "AI Health" tab that pulls from Langfuse via its SDK: total AI queries this month, average faithfulness score, average officer feedback score, average response latency, which schemes are being looked up most frequently by ward, and cost trends. This makes AI spending transparent and gives the Commissioner a single view of whether the AI is performing correctly across all wards.

How the Two Repos Connect
The RAG-System repo's retrieval layer moves into JanVedha's backend as a new service. The generator (Ollama) is discarded entirely and replaced by the Gemini client already in use in JanVedha. The existing RAG-System tracer (which writes JSON to a local file) is also discarded and replaced by direct Langfuse SDK calls. The scheme advisor has its own separate ChromaDB collection so it does not interfere with anything else. The Langfuse client is initialized as a singleton at application startup and used by both the scheme advisor and — going forward — by the existing chat, classifier, and priority agent features in JanVedha.

What the User Sees
Councillors get a new page in their dashboard: a text area to describe a constituent, their ward pre-filled, a single button, and a structured results panel organized into eligible/partial/ineligible sections with scheme cards, document checklists, application guidance, source citations, and feedback buttons. Query history is accessible so they can look back at past assessments. Commissioners see the AI Health tab with usage and quality metrics. The Langfuse UI itself is for the technical team only — the government-facing dashboards show curated summaries, not raw traces.
